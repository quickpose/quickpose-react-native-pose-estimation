QuickPose React Native

User-configurable props, features, layout, and event data

Quick Start

import { QuickPoseView } from '@quickpose/react-native';

<QuickPoseView
  sdkKey="YOUR_SDK_KEY"
  features={['overlay.wholeBody']}
  useFrontCamera={true}
  style={{ flex: 1 }}
  onUpdate={(event) => {
    const { results, feedback } = event.nativeEvent;
    // results: [{ feature: string, value: number }]
    // feedback: string | null
  }}
/>

Props

PropTypeDefaultDescription
sdkKeyrequired string — Your QuickPose SDK key from dev.quickpose.ai
featuresrequired string[] — Array of feature strings to detect. Can be updated at runtime — the camera stays running and features hot-swap without restart.
useFrontCameraoptional boolean true Whether to use the front-facing camera.
styleoptional ViewStyle — Standard React Native view style. Controls the camera view size and position. Use flex: 1 for fullscreen, or set explicit width/height for a smaller viewport.
onUpdateoptional (event) => void — Called every frame with detection results and feedback. See Event Data below.

Layout & Sizing

The QuickPoseView is a standard React Native view. You control its size and position with the style prop — the native camera preview and skeleton overlay fill the view bounds automatically.

Your Container View
QuickPoseView (style prop)
Camera Preview
+ Skeleton Overlay
Your own UI (buttons, text, etc.) goes outside or absolutely positioned on top

Common Layout Patterns

Fullscreen Camera
style={{ flex: 1 }}
Fixed-size Camera Window
style={{ width: 300, height: 400, borderRadius: 12 }}
Camera with UI Below
<View style={{ flex: 1 }}>
  <QuickPoseView style={{ flex: 3 }} ... />
  <View style={{ flex: 1 }}>
    {/* Your controls, counters, etc. */}
  </View>
</View>
The overlay (skeleton lines, joint angles, etc.) is rendered natively by the SDK and fills the camera view automatically. You don't need to position or style the overlay separately.

Event Data (onUpdate)

The callback receives an event with event.nativeEvent containing:

FieldTypeDescription
results { feature: string, value: number }[] One entry per feature in the features array. The value meaning depends on feature type — see below.
feedback string | null User guidance text from the SDK, e.g. "Move further away" or "Keep your arm straight".

Value Interpretation by Feature Type

Feature TypeValue RangeMeaning
Overlay0 or 11 = body detected, 0 = not detected
Range of Motion0–360Joint angle in degrees
Fitness (reps)0.0–1.0Exercise progress (0 = start position, 1 = end position). Count reps by tracking 0→1→0 cycles.
Fitness (plank)0.0–1.0Hold quality (1.0 = perfect form)
Raised Fingers0–5Number of raised fingers
Thumbs Up/Down-1, 0, 1-1 = down, 0 = neutral, 1 = up

Input / Gesture

Hand gesture recognition.

raisedFingers
Count raised fingers (0–5). Optional: raisedFingers.left / .right
thumbsUp
Detect thumbs up (value: 1). Optional: thumbsUp.left / .right
thumbsUpOrDown
Detect thumbs up (1) or down (-1). Optional: thumbsUpOrDown.left / .right

Multiple Simultaneous Features

Pass multiple feature strings to detect several things at once. Each gets its own entry in the results array:

features={[
  'overlay.upperBody',
  'rangeOfMotion.shoulder.right',
  'fitness.bicepCurls',
]}

// onUpdate results:
// [
//   { feature: 'overlay.upperBody', value: 1 },
//   { feature: 'rangeOfMotion.shoulder.right', value: 45.2 },
//   { feature: 'fitness.bicepCurls', value: 0.73 },
// ]

Platform Setup

iOS
Add NSCameraUsageDescription to Info.plist. Minimum deployment target iOS 15.0.
Android
Camera permission is requested at runtime by the SDK. Minimum SDK 24.
